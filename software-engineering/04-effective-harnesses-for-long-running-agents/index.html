Initializer 四项交接资产

第一轮结束前必须交付，后续轮次才能稳定接力

01
📜
启动脚本
init.sh

一键启动环境和验证流程，避免重复配置

02
📝
进度日志
claude-progress.txt

记录进度和下一步建议，减少接手成本

03
📌
代码基线
git commit

提供基线，便于回滚和比较

04
📋
验收清单
feature list (JSON)

显式“完成定义”，防止 Agent 凭感觉结束任务

⚠ Agent 只能改 passes 字段，禁止删改条目——这是防止“偷改清单蛒混过关”的强约束
关键点：feature list 不是普通待办，而是防止“凭感觉结束”的验收锚点

Coding Agent 增量纪律

每轮只做一个“最小增量”，用工程化节奏约束 Agent 的执行边界

1
READ

读取状态

查看 `pwd`、进度记录、feature list、git log

2
BUILD

实现一件事

只处理一个优先级最高的 feature，避免“一次做太多”

3
VERIFY

本轮验证

做完立刻验证，确保环境持续可运行、可交接

4
LOG

更新进度

记录已完成、待完成及下一步建议

5
COMMIT

提交交接

git commit 带描述，保持环境干净

Getting up to speed · 每轮启动序列
pwd 确认目录
→
读 git log + progress
→
选最高优先 feature
→
跟 init.sh 启动服务
→
基线验证通过再开始

从“感觉完成”到端到端验收

单测和接口冒烟不够，必须按真实用户方式跑完整路径

❌ 感觉完成

  • 代码运行，局部测试通过
  • 接口响应正常
  • 用户路径未完整验证
➔

✅ 验收完成

  • 浏览器自动化完整流程
  • 交互和状态变化正确
  • 用户视角路径真正跑通
⚠ 已知局限：Claude 目前无法通过 Puppeteer MCP 看到浏览器原生 alert 弹窗，依赖 alert 的功能需要额外关注

未来方向

原文指出的两个开放问题，也是下一步值得关注的方向

🧩 单 Agent vs 多 Agent 架构

现有方案由一个通用 Coding Agent 串联所有上下文窗口。但专职化的 Testing Agent、QA Agent、代码清理 Agent 是否能在各子任务上表现更好？这个问题目前尚没有明确答案。

🌍 迁移到 Web 之外

现有 Harness 针对全栈 Web 应用优化。这套方法能否推广到科学研究、金融建模等其他长周期 Agent 任务场景？原文认为有很大潜力。

关键洞察：不要把现在的 Harness 设计当成终态——它是现阶段的最佳实践，下一阶段的更多可能性已在路上。

一句话总结：工程化框架决定上限

"Agent 想稳定交付长任务，
决定上限的不是模型性能，
而是工程化 Harness。"

Anthropic 的实践验证了这一点：
交接资产保障起点，增量节奏控制过程，端到端验收定义终点——
三层机制齐备，长任务才真正可控。

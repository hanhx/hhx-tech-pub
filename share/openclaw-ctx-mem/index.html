ContextEngine 到底开放了什么？

v2026.3.7 核心架构升级 —— 从"硬编码记忆"到"可插拔引擎"

第一部分
顶层结论
ContextEngine 不是一个"新记忆功能"，而是把"上下文管理"从底层固定逻辑， 变成了一套可替换的插件接口。

痛点与旧方案

  • 以前的尴尬：记忆处理写死在 Core 里，满载就直接粗暴截断，AI 经常"失忆"。
  • 开发者困境：想优化记忆策略，只能去改底层源码，成本高、风险大、升级也麻烦。
  • 结果：官方方案再通用，也很难覆盖无损记忆、RAG 检索、长周期项目等复杂场景。

核心架构变化

  • 开放接口：官方开放了 7 个生命周期钩子，让上下文处理从"内置逻辑"变成"标准接口"。
  • 默认兜底：内置 LegacyContextEngine，老用户不改配置，行为完全一样。
  • 关键意义：OpenClaw Core 只负责调用规范，不再强绑定唯一记忆实现。
Bootstrap Ingest Assemble Compact AfterTurn Sub-agent Lifecycle

带来的结果

  • 一秒热切换：现在只需在 config 文件里改一行配置，就能不改源码地替换整套记忆策略。
  • 社区创新：如果你觉得默认实现不够强，可以直接接入更激进的插件实现。
  • 爽点：比如切到 lossless-claw，就能把"会失忆"升级成"尽量无损记忆"。

🎯 一句话带走

ContextEngine 的价值，不是"官方又加了一个记忆功能"， 而是把"记忆怎么做"这件事，正式开放给了开发者生态。
不满意官方方案？不用再改 Core，直接拔掉默认实现，换一个更适合自己业务的记忆引擎。

lossless-claw 凭什么敢叫无损记忆？

ContextEngine 插件实现 —— 接管 7 大接口，实现按需无损的记忆管理

第二部分
🎯 结论：作为 ContextEngine 的插件实现，它通过接管 7 大生命周期钩子，用状态机将记忆打包成极高的 信息抽象度 (DIA)， 构建了一本"带隐藏超链接的 DAG 动态目录"，让模型主动向下钻取，实现真正的 0 损失。

❌ 传统方案的痛点

  • 滑动窗口（直接删）：
    做法：满载后像切香肠一样截断。
    后果：抛弃底层逻辑，AI 彻底断片。
  • 传统 RAG（容易幻觉）：
    做法：把代码切碎算成向量盲搜。
    后果：代码逻辑有严格因果，RAG 破坏了原始上下文，经常"找错行"。

⚙️ 核心架构：接管 7 大接口盘出 DAG 树

1. bootstrap
启动时建联 SQLite，加载旧树
2. ingest
拦截每句话，全量存入底层库
3. assemble
组装 "原话" + "顶层摘要" 发送
4. compact
快满时触发，高 DIA 提炼建树
5. afterTurn
对话结束刷新游标与缓存
6. subagent
开分身时分配隔离的沙盒
7. lifecycle
关机前确保待建摘要落盘

✅ 带来的爽点（按需展开）

  • 结构化路由：
    机制：发给模型的精简摘要里，暗藏着指向底层 SQLite 数据的锚点。
  • AI 亲自翻抽屉：
    体验：模型遇到要查证的变量，会调用工具顺着链接"降维展开"原始代码。
  • 双赢：既不爆 Token，又保住细节！
plugins:
  slots:
    contextEngine: "lossless-claw"
💡 核心对比：为什么不用传统的 RAG？
传统滑动窗口 像 碎纸机。抽屉满了就把最早的历史切碎扔掉，前置细节彻底断片。
传统切块 RAG (Naive) 像 搜索结果页。把代码切碎靠相似度盲搜，丢失了严格的因果与作用域，极易缝合报错。
lossless-claw 像 带目录的内部 Wiki。先看高层摘要，模型觉得缺细节时，顺着锚点主动下钻展开原文。

🎯 一句话带走

"真正的无损，不是把几十万字硬塞给模型，而是给模型建一本带超链接的目录。"
利用 DIA 理论提升摘要的抽象度，用 DAG 树锁死代码的因果关系。这种降维展开的思路，是 Vibe Coding 时代的最佳解法。

AI Agent 完整记忆架构设计

借 OpenClaw ContextEngine 分层，lossless-claw 无损，OpenViking 跨会话，构建独立记忆系统

第四部分

1. Session Runtime / Agent Core

职责：只"调用"不"实现"
  • 接收用户输入
  • 调用 ContextEngine.assemble()
  • 调用 MemoryEngine.recall()

2. ContextEngine（会话内无损记忆）

职责：当前会话的摘要树管理
  • 两层：Raw Layer（全量存原始消息）
  • DAG Layer（L0→L1→L2→L3 摘要树）
  • 对模型只发"摘要 + 锚点"

3. MemoryEngine（跨会话 / 人员记忆）

职责：长期记忆的提取、存储、召回
  • 类似文件系统 /memory/users/.../projects/.../agents/...
  • 用"路径粗筛 + 向量精炼"做检索

4. Storage & Index（底层）

  • Raw Message Store
  • DAG Node Store
  • Knowledge Index（向量 + 路径）
本层"数据底座"是上层共享的底层存储资源，可选择 SQLite、本地文件、向量数据库等不同实现，而不影响上层插件接口。

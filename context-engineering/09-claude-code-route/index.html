层一、层二：时序与类型控制

Claude 正忙时输入不会丢 · 有些命令可以插队

层一 · onSubmit
⚡

Immediate 命令插队

标记了 immediate: true 的命令（如 /btw），无论 Claude 是否正忙，直接执行，不进队列。专为"需要打断"设计。

📬

队列缓冲，不丢失

Claude 正忙时，非 immediate 输入调用 enqueue() 进 messageQueueManager。顺序保留，等 Claude 完成当前任务后依次消费。

层二 · handlePromptSubmit
🚪

退出关键词短路

exit / quit / :q / :q! / :wq——直接触发 exit 命令，不进模型，也不进队列。熟悉 vim 的用户可以直接退出。

🖥

local-jsx immediate 本地UI

/config、/doctor 这类命令在 Claude 正忙时也能执行：渲染本地 React JSX 界面，完全绕过模型。等 onDone 回调触发后自动清除。

以上都不满足 → executeUserInput() → 进入层三

层三：内容路由的四条出口

processUserInputBase · 顺序匹配 · 命中即返回

出口一 · 语义关键词
ultraplan 关键词
输入含 ultraplan 相关词汇，且非斜杠前缀、非 headless 模式。

检测在 preExpansionInput（粘贴占位符展开前）进行，防止粘贴内容误触发。

输入被重写为 /ultraplan [原文]，走斜杠命令分支。
→ processSlashCommand
出口二 · 输入模式
mode === 'bash'
用户用 ! 前缀触发的 bash 模式。

本地执行 shell 命令，结果作为工具输出注入上下文。

模型不参与命令生成，只看到执行结果。
→ processBashCommand（本地）
出口三 · 前缀匹配
input.startsWith('/')
斜杠命令，内部再分支：

大部分（/clear、/help、/model）本地执行，不消耗 token。

少数（/commit）内部会调用模型生成内容。

斜杠命令自己决定要不要进模型。
→ processSlashCommand（部分进模型）
出口四 · 兜底
其余所有输入
前三条都不命中，普通自然语言。

这里才是上下文工程流水线的真正入口：输入被包装成 UserMessage，附件收集，然后进入 09 期讲的那套加载流程。
→ processTextPrompt → 上下文工程
四条出口按顺序判断，命中即返回，不继续往下匹配。这意味着如果你的自然语言输入碰巧以 / 开头，它会走斜杠命令分支，而不是进模型——这是边界情况，也是为什么斜杠命令不能包含歧义词。

多意图输入：路由层无感，系统三招保住

"帮我重构、补测试、提 PR" → 一条消息 → 三个意图打包进模型

路由层不做意图解析。三个意图打包在一条普通文本里，走第四条出口，原封不动送进模型。分意图是语言模型的能力，不是路由层的职责。
策略一 · 最简单

顺序执行

模型在 while(true) 循环里一个一个来，原始消息始终在对话历史里。

⚠ 风险：上下文变长后，原始消息被 autocompact 压缩，意图可能丢失。
策略二 · 系统保障

TodoTool

模型调用 TodoWrite 创建结构化任务清单。附件系统每轮注入当前状态：

maybe('todo_reminders', () =>
  getTaskReminderAttachments(
    messages, toolUseContext
  )
)
系统每轮主动提醒模型还有哪些任务未完成。
策略三 · 最结构化

PlanAgent 子 Agent

主 Agent 调用内置的 PlanAgent——只读"软件架构师"。它先 Glob/Grep 探索代码库，再设计方案、列出 3-5 个关键文件，以文本返回。主 Agent 拿到结果后调用 TodoWrite 写任务清单。

规划阶段完全只读，不改任何文件。≠ Plan Mode（那是另一个工具）。

三个值得单说的工程细节

CCR 远程桥接 · 图片并行 · queryGuard 互斥锁

📱

远程桥接（CCR）另一套规则

从手机/浏览器远程发来的输入带有 bridgeOrigin 标记，skipSlashCommands 默认为 true。

远程输入里的 /config 不会触发本地 UI，被当普通文本处理。只有 isBridgeSafeCommand 白名单里的命令例外。

防止远程客户端触发只适合本地终端的操作。
🖼

图片压缩与路由并行

粘贴图片时，maybeResizeAndDownsampleImageBlock 在路由判断之前就并发跑起来。

图片处理（可能耗时）不阻塞路由逻辑。两件事同时进行，图片处理完了，路由结果也到了，合并后继续。

减少等待时间，I/O 密集型操作不堵塞控制流。
🔒

queryGuard 互斥锁

整个系统保证同一时刻只有一个 executeUserInput 在运行。

queryGuard.reserve() 在进入执行前锁定，onQuery 的 finally 块释放。队列机制 + queryGuard 共同保证并发输入不会造成两个 query 同时运行。

防止竞态条件，保证会话的线性一致性。

理论基础

这套机制背后的三层理论支撑

设计模式

责任链模式

Chain of Responsibility——每一层处理自己能处理的，处理不了的往下传，直到最后一层兜底。Claude Code 的三层就是一条责任链，只是最后那层的"兜底"是交给语言模型。

工程原则

最小代价原则

能本地确定性解决的问题，不应走模型。/help 输出固定，bash 命令执行确定，没有理由花 token 和延迟换一个确定性答案。所有混合系统（规则 + AI）的基本立场。

学术框架

Neurosymbolic AI

规则系统和神经网络各司其职，边界在于问题的确定性程度。路由层是"符号系统"入口，模型是"神经网络"出口。输入越确定，越应在路由层短路；越开放，越应进模型。

CE 09（这期） 是上下文工程的入口判断：哪些输入有资格进流水线、什么时候进。
CE 10（下期） 是流水线本身：进了之后，模型能看到什么。

两期合在一起，才是从"你按下回车"到"模型开始推理"的完整链路。

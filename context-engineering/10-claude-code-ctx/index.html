阶段二：加载初始上下文

三份上下文并发加载 · 后台记忆预取同步启动

SYSTEM PROMPT

核心系统指令

告诉模型"你是 Claude Code，你有哪些工具，该怎么用"。包含所有工具的定义和使用规则。

整个会话期间不变 → prompt cache 的稳定前缀
USER CONTEXT

用户级上下文

核心是 CLAUDE.md 的内容：项目结构、编码规范、架构约束。还包括当前日期。

会话启动时一次性加载
CLAUDE.md 不存在 → 跳过
--bare 模式 → 跳过
SYSTEM CONTEXT

系统级上下文

主要是 git status 快照：当前分支、最近 5 条 commit、工作区变更状态。

⚠ "snapshot in time, will not update during the conversation"——启动时拍快照，后面不更新。
三者并发加载（Promise.all）
通过 appendSystemContext 拼接，构成完整系统提示。

此时模型上下文 =
系统指令 + CLAUDE.md + git 状态
+ 你的输入 + 首轮附件
同步启动：startRelevantMemoryPrefetch
在后台用 Sonnet 搜索与输入相关的记忆文件，不阻塞主流程。

模型开始推理时，记忆搜索已经在跑了。
结果在 while(true) 循环的第一轮被消费。

小结：Plan 阶段上下文就绪

这两步完成之后，模型才被调用

PLAN 阶段能做好规划

CLAUDE.md 提供项目规范
git status 提供当前分支状态
@文件 + 首轮附件提供具体代码

信息够了，规划才能落地。

EXECUTE 阶段的上下文起点

这两阶段加载的上下文是整个 Execute 循环的基础。while(true) 每一轮在此之上叠加新信息（附件注入），而不是每轮重新加载。

Prompt Cache 设计

systemPrompt 整个会话不变 → 稳定前缀，始终命中缓存。CLAUDE.md 一次性加载 → 不在每轮重读。两者都是有意识的缓存设计。

按下回车之后，模型还没开口前，
系统已经完成了输入解析、首轮附件收集、钩子执行，
以及系统指令 + CLAUDE.md + git 状态的并发加载，
并在后台悄悄启动了记忆搜索。

这两个阶段，是 Plan-and-Execute 里 Plan 阶段能做好规划的上下文基础。

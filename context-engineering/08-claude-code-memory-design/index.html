第一层：存得准 ① 存什么

四类记忆约束 · 只存不可派生的上下文

memoryTypes.ts 注释第一行：只能存无法从当前项目状态派生出来的上下文

👤

user

用户的角色、目标、背景知识。Go 老手初次接触 React → 下次解释用后端类比

🔒 永远 private

💬

feedback

纠偏 + 被确认的做法。只记错误会让模型越来越保守

默认 private，项目规范存 team

📋

project

进行中的目标、deadlines、incidents。相对日期转绝对日期

🔥 强烈推荐 team

🔗

reference

外部系统指针：Grafana URL、Linear 项目名、Slack 频道

通常 team

排除项：代码模式、架构决策、git 历史、文件结构、CLAUDE.md 已有内容、临时会话状态——这些 grep/git 可查的，一律不能存。即使用户显式要求也不行。

第一层：存得准 ② 存的机制

MEMORY.md 索引设计 · Topic File Frontmatter

MEMORY.md — 索引而非内容

每行一条指针，一行摘要。严格控制在 200 行 / 25 KB 以内

# 这是索引，不是内容
- [用户背景](user_role.md) — Go 老手，React 新手
- [测试规范](feedback_testing.md) — 禁 mock DB
- [发布冻结](project_freeze.md) — 2026-04-03 起
- [监控看板](ref_grafana.md) — latency oncall 面板

⚠ 超出截断 → 索引失效 → 召回断链。每次会话启动自动注入 system prompt

Topic File — frontmatter 三字段

---
name: 用户背景信息
description: Go 老手，初次接触 React，解释时用后端类比
type: user
---

name — 文件标题，保持与内容同步

description — 给 Sonnet 读的，描述越精准，召回越准

type — 四类枚举之一，决定 scope 推荐

两步写法：先写 topic file → 再在 MEMORY.md 加一行指针。更新优先于新建

存储路径：~/.claude/projects/<sanitized-git-root>/memory/ — MEMORY.md 索引 + topic files + team/ 子目录（TEAMMEM 开启时）

3

后台提取

extractMemories 子代理
主模型产出最终回复且无工具调用时触发，主代理写了就跳过
完整 fork 共享 prompt cache，几乎零成本启动
2 轮高效写法：先并行 Read → 再并行 Write
工具受限：仅 Read/Grep/Glob + Write/Edit（memory 目录内），Bash rm 禁止

🤖 为什么用 Sonnet 做召回？

召回是一次 sideQuery（旁路轻量调用），独立于主对话。主模型可能是 Opus，但选哪 5 个记忆文件只是读 description 做匹配，用 Sonnet 成本低、速度快，没必要占主模型资源。

输入：所有 topic file 的 [type] filename: description 列表 + 当前 query；输出：JSON schema 结构化的 selected_memories 数组。超 1 天的文件注入时自动附 staleness 警告。

第三层：维护不过期 ① autoDream

前两层管了写和读——记忆会腐化，这是第三层的使命

三个角度：① 主动整合  ·  ② 协作隔离  ·  ③ 读时预警  |  本页：① autoDream · DreamTask

为什么需要它？记忆有熵增

代码库在变
文件路径可能失效
项目状态在变
deadline 早就过了
用户在进化
旧纠偏可能反而变错
description 重叠
召回精度持续下降

没有 autoDream，记忆只是只增不减的日志堆

触发三道门 源码注释按成本从低到高排

①
时间检查　距上次整合 > 24h　一次 stat，最便宜
②
会话计数　累计完成 ≥ 5 个会话
③
分布式锁　防多进程同时跑
P1

Orient 定向

ls 记忆目录 + 读 MEMORY.md 全文
找孤儿文件（有 topic file 但索引无指针）

P2

Gather 采集

扫所有文件 frontmatter + 正文
标记漂移：description 对不上 / 相对日期失效 / 路径不存在

P3

Consolidate 整合

合并重叠文件，重写 description
过期 project 记忆降级不删，标记为历史

P4

Prune 剪枝

重写 MEMORY.md，去过时条目
确保总行数 ≤ 200 行

KAIROS 长时模式：切 append-only 日志写法（防数据竞争——主代理还在跑，直接改 topic file 有风险），每天写 logs/YYYY/MM/YYYY-MM-DD.md，夜间 /dream skill 蒸馏归档。

第三层：维护不过期 ② 团队记忆 · 新鲜度

双目录隔离 · staleness 自动警告

👥

团队记忆（TEAMMEM）

开启后变成双目录架构

private/
用户独享
跨会话持久
team/
贡献者共享
每次会话同步
user → 永远 private
feedback → 个人 private / 规范 team
project → 强烈推荐 team
reference → 通常 team

🚫 team 记忆禁止存 API key 或凭证

⏱

新鲜度管理

memoryAge.ts — 超过 1 天自动附警告

"This memory is N days old. Memories are point-in-time observations, not live state — claims about code behavior or file:line citations may be outdated. Verify against current code before asserting as fact."
记忆提到文件路径 → 先 check 存在
记忆提到函数/flag → 先 grep 验证
"记忆说 X 存在" ≠ "X 现在存在"

今天/昨天 → 不附警告  |  ≥ 2 天 → staleness 提示

记忆系统三层闭环

🎯

存得准

四类记忆约束：user · feedback · project · reference
只存不可从代码/git/CLAUDE.md 派生的认知
每种类型内置 scope 推荐 + body 结构规范

🔍

取得准

MEMORY.md 索引 200行/25KB 控制加载量
description 字段决定召回质量（由 Sonnet sideQuery 读取，非主模型）
alreadySurfaced 防重选 · extractMemories 自动提取

🔄

不过期

autoDream 4 阶段周期蒸馏
团队双目录按类型推荐 scope
memoryAge staleness 警告，用前先验

"不是让模型'记住一切'，
是让模型'记对该记的'。"

存得准靠四类记忆约束，
取得准靠启动静态加载 + Sonnet 动态召回 + 后台自动提取子代理，
不过期靠 autoDream 蒸馏 + 新鲜度验证。

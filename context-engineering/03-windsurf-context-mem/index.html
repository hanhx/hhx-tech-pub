一张图讲清楚：商用 Windsurf 是如何做到「既记得住，又不乱看」的？

架构对标解析 03 · Memory × Context
基于公开资料与社区技术文章整理，仅为架构抽象示意
🎯 顶层结论：
Windsurf 接受「有控制的遗忘」，用记忆模块决定长期要留什么，用上下文引擎决定这一步要看什么。
lossless‑claw 则尽量「不遗忘」，靠全量归档 + DAG 保证以后随时还能把原文翻回来。

① 记忆生成：长期要留下什么？（不囤原文）

  • 自动记忆： 当对话里出现架构决策、约束条件、命名约定时，Cascade 会在后台生成一条结构化 Memory，挂在当前 Workspace 下面。
  • 主动记忆： 手动把某条信息钉进 Workspace 级的记忆库，不完全依赖模型自己判断该记啥。
  • 记忆隔离： 按 Workspace 隔离、本地存储，每个工程有自己的一组 Memories / Rules，存在本地配置目录，不进仓库、不串项目。

② 记忆读取：这一轮要用哪几条？

  • 结合语义 + 工程状态算相关性： 看这一轮的提问、光标位置、当前文件、报错栈等，对所有 Memories / Rules 做相关度打分。
  • 只留下少量高相关摘要： 不把整库记忆都塞回去，只保留和当前任务强相关的那几条，形成一个「记忆候选集」。
  • 只负责「选料」，不负责「怎么用」： 这一层只决定用哪些记忆，至于是进 System Prompt、还是当作工具结果，留给下一步 RAG 上下文引擎来安排。

③ RAG 上下文引擎：Local Index + 代码 + 记忆 → 模型真正能看到的那 1%

1. What Context：先看人现在在干嘛 观察光标、当前文件、Stack Trace、最近编辑、终端输出，先划定「大概在哪一片代码区域找答案」。
↓
2. Fast Context（RAG 检索 · Local Index） 在为整个仓库构建好的 Local Index 上，由 Fast Context 子代理（SWE‑grep / SWE‑grep‑mini）发起多轮、并行的代码检索，从代码库中快速筛出一批候选文件和代码片段。
↓
3. SWE‑1 Swarm：调用链分析 + 语义重排 沿调用链和依赖关系分析，再结合语义相似度，把大量「可能相关」的候选片段压成那 1%「非看不可」的代码、配置，并把②给出的少量记忆摘要一起打包成上下文候选。
↓
4. Frontier 模型：在精简上下文里做最终决策 Claude Opus 4.6 / SWE‑1.5 只在这 1% 高信息密度上下文里做规划、重构、解释、找根因，而不是亲自从零翻整仓库和整库记忆。

粗暴理解：先由本地索引 + 小模型 + 中间层模型按「目的」帮你选好代码 + 记忆，再让大模型在精简上下文里动昂贵算力。

💡 终极对比：lossless‑claw 把「上下文 = 记忆」，Windsurf 把「记忆」和「RAG 上下文」拆开配合用
Lossless-claw（全量归档 · 开源） 路线：上下文 = 记忆 = 历史——所有对话原文都写进 SQLite，用 DAG 把因果串一遍，后面靠工具帮模型「翻档案」。
适用：需要「可审计 / 可迁移 / 可回放」的长期项目档案。
Windsurf（记忆 + RAG 上下文解耦 · 商业） 路线：记忆模块负责「存摘要」，RAG 上下文引擎负责「在 Local Index 上检索 + 重排代码片段，并按需注入少量记忆」，最后交给 Frontier 模型思考。
适用：追求「顺手体验 + 低延迟 + 成本可控」的一线开发场景。
从左到右，就是 Windsurf 这一条线：先决定记什么 → 每一轮从记忆里挑什么 → 再在 Local Index 上用 RAG 选代码 + 注入记忆 → 一起喂给大模型。

🎯 一句话带走

lossless‑claw 想的是「别丢任何一段历史」，Windsurf 想的是「每一步只让模型看到最有用的那一点点上下文」。

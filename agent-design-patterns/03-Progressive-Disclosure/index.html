AI Agent 设计模式 · 03

渐进式披露

能力很多，为什么不能一开始全塞进上下文？

这就是渐进式披露要解决的问题。
Claude Code 的 Skills，只是这套模式最典型的工程样本。
全部预加载，上下文会被撑爆；完全不加载，模型又不知道自己有什么能力。
所以真正有效的做法不是全给，也不是不给。
而是先少给，再展开，再收回。

另一类 Agent 模式

不是推理—行动模式，是上下文工程模式

推理—行动模式

解决的问题：下一步做什么？

ReAct → Thought-Action-Observation 循环

Plan-and-Execute → 先规划再执行

Reflection → 自我批评、迭代修正

关注点：推理策略、行动路径

vs

上下文工程模式

解决的问题：行动之前，哪些信息先进上下文？

上下文里该有什么？ → 粒度和范围

何时展开细节？ → 按需加载时机

用完了怎么收？ → 回收和压缩策略

关注点：信息编排、注意力预算

渐进式披露属于后者。
它不告诉 Agent 下一步做什么，而是决定 Agent 行动之前，信息以何种粒度、何时进入上下文。

概念源头与工程语境

一条 HCI（人机交互）原则 + 一个工程约束

📄 HCI 概念来源（人机交互）

先摘要，后展开

Springer & Whittaker (2018) 发现：一次性暴露所有细节，用户判断反而更差。先给简化摘要、按需展开细节，判断更准。

把"用户"换成 Agent 的注意力机制，这条原则就在 Agent 语境里站住了。

arXiv:1811.02164 · Progressive Disclosure: Designing for Effective Transparency
⚙️ 工程语境

Context Rot：上下文腐化

Anthropic 指出：上下文窗口是有限的注意力资源。Token 越多，模型提取信息的能力越弱——信息不是消失了，而是被稀释到模型注意不到。

解法：Agent 只维护 轻量级标识符，在需要时才动态加载数据，不预先填满上下文。

Anthropic · Effective Context Engineering for AI Agents · 2025.09
一条 HCI 原则提供了"先摘要后细节"的认知基础；
一个工程约束——注意力预算有限，不能一开始全塞满——让它成为必须。

Skills：四步闭环

Skills 可以有几十上百个，每个最多 5K tokens。全部预加载上下文会爆，完全不加载模型不知道自己有什么能力。
解法是分层加载——暴露入口 → 按需展开 → 执行后收折 → 压缩后选择性保留。

① 暴露入口
~100 tokens

轻量元信息

只注入技能名称和触发条件。模型知道有什么能力，但看不到具体内容。

→
② 按需展开
~5K tokens

加载完整正文

匹配到任务后主动调用，完整技能内容才进入上下文。不调用就不进来。

→
③ 执行后收折
~200 tokens

剥离正文

执行完成，详细内容从上下文剥离，只留必要的结果摘要。

→
④ 压缩后保留
用过的

选择性重注入

优先保留真正用过的内容，完整列表不再灌回去。

① 暴露入口 源码观察 
// 会话启动 → 扫描所有 .md，只解析 YAML 头
const skills = await scanSkillFiles(skillDirs)
const listing = skills.map(s => ({
  name:        s.frontmatter.name,
  description: s.frontmatter.description,
  // 正文不加载，只取元信息
}))
// 注入 system prompt → 模型只看到目录
② 按需展开 源码观察 
// 模型主动调用 → 才加载完整正文
async execute({ skillName }) {
  const content = await readFile(skill.filePath)
  return truncate(content, MAX_SKILL_CONTENT_LENGTH)
  // 不调用 → 正文永远不进上下文
}
③ 执行后收折 源码观察 
// 执行完成 → 完整正文从上下文剥离
return {
  role: 'tool_result',
  content: summarize(result, MAX_SUMMARY_LENGTH),
  // 5K tokens → 200 tokens，用完即收
}

结论：渐进式披露到底是什么？

它不是一个 Prompt 技巧，也不是一个新术语。
它是 Agent 在有限上下文窗口里，管理大规模能力系统的一种工程方法。

📄 论文层

先摘要，后展开。
信息不是越多越好，粒度对了才有效。

🔬 工程研究层

上下文是有限的注意力资源。
塞得越多，提取越弱——这就是 context rot。

⚙️ 工程层

先暴露入口，按需展开，用完收折。
Skills 把这个原则做成了四步闭环。

从 HCI 的认知原则，到 context engineering 的工程约束，再到 Claude Code Skills 的实现样本——
渐进式披露不会让模型更聪明。它只是让有限的注意力，用在更重要的地方。

ReAct 的局限

规划和执行是两种不同的认知任务

❌ ReAct 在长任务的困境

全局目标漂移
做着做着忘了最初要干嘛，在某个细节里越陷越深，全局目标早就丢了。

认知模式冲突
规划需要全局视角、延迟判断；执行需要聚焦当前、快速行动。让同一个循环同时承担这两件事，本身就是矛盾的。

➔

✅ Plan-and-Execute 的解法

三个阶段各司其职
Plan 阶段列出任务清单；Execute 阶段逐步执行；Re-plan 重新评估是否需要调整。

计划可检查、可约束
Plan 阶段的输出要足够具体，让 Execute 阶段照着走、不会走岔。计划生成后可以插入人工审核，批准后才执行。

🗺️

Plan 阶段

接收用户目标
列出详细的任务清单
定义依赖与完成标准
不下场执行工具

⚡

Execute 阶段

接收 Plan 的任务清单
翻译为工具调用与操作
实时推进执行
不操心整体走向

🔄

Re-plan

每轮执行后重新评估
任务完成？计划还对？
决定是否继续或调整
与 ReAct 的根本区别

LangChain：模式定义

plan-and-execute agents · 工程实践的归纳

核心流程三步：Planner 先列出多步任务清单 → Executor 逐步执行 → 系统触发 Re-planning。Planner 再次判断：任务完成了就结束，还没完就生成新的任务清单，再跑一轮 Execute。
🗺️

① Planner 生成计划

接到用户目标后，输出结构化任务清单：先做什么、后做什么、依赖关系、完成标准。Planner 不负责动手——它只管"想"。

⚡

② Executor 逐步执行

拿到任务清单后，Executor 逐步执行——把每一步翻译成工具调用和具体操作。Executor 不操心全局走向。

🔄

③ Re-planning 循环

执行结束后，系统主动触发 Re-planning。Planner 再次判断：任务完成了就结束，还没完就生成新的任务清单，再跑一轮 Execute，如此循环直到完成。

ReAct 是"边想边做，走哪算哪"；
Plan-and-Execute 是"先想清楚，做完再看是否要重新想"。

Claude Code：源码落地

Plan · Execute · Re-plan 三阶段落地

🗺️

① Plan 阶段

模型调用 EnterPlanMode，权限切到 plan。权限一切换，系统就会拒绝其他文件的写操作——这是强制约束，不是靠模型自觉。模型只能探索代码库、设计方案，产出是一份计划文件。

⚡

② Execute 阶段

模型调用 ExitPlanMode 提交计划，用户审批后权限恢复。模型照着计划文件开始改代码——写操作重新可用。

🔄

③ Re-plan

Re-plan 不是 Execute 之后才触发的独立步骤，而是嵌在 Execute 里的能力。模型在执行过程中持续参照计划文件，它是“接下来做什么”的唯一依据。发现计划有问题，直接改文件就行——不需要等执行结束，也不需要系统触发。

底层：三个阶段跑在同一个 queryLoop（while(true)）里，靠一个权限值划边界，靠一个文件做交接。

context.setAppState({ toolPermissionContext: { mode: 'plan' } })
// 写操作工具检查权限 → 拒绝执行
// 模型探索代码库、写计划文件
⚡ Execute：ExitPlanMode 用户审批后恢复 
context.setAppState({ toolPermissionContext: { mode: prePlanMode } })
// mode 恢复到进入前的状态，写操作重新可用
// 计划文件随时可改 → Re-plan 无需额外机制

两种实现对比

同一个模式，不同的工程选择

📦 LangChain 路径

架构分离

Planner 和 Executor 是两个独立组件

Executor 内部跑自己的 ReAct 循环

Re-planning 由系统主动触发，Planner 重新评估

两个角色 · 两套循环 · 一个正式 Re-planning 入口

vs

⚙️ Claude Code 路径

状态切换

同一个 while(true) 循环 跑全程

靠 toolPermissionContext.mode 区分阶段

Re-plan 嵌在 Execute 里——模型随时直接改计划文件

一个循环 · 权限值变化 · 计划文件串联

Claude Code 的 Plan Mode 也是 Plan-and-Execute——先规划、后执行、中间有审批边界，三条全满足。
只是实现手法不同：不拆组件，靠权限状态在同一个 while(true) 循环里跑完 Plan、Execute、Re-plan 三个阶段。
有意思的地方在于：这个循环本身是 ReAct 的机制实现。Plan-and-Execute 是在它之上，靠权限状态和计划文件构建出来的模式。同一个循环，不同的抽象层级。

结论

先规划、再执行、随时能重新评估，
是让 Agent 真正走完长任务的核心设计。

📦 LangChain 路径

架构分离——Planner 和 Executor 各管各的，Re-planning 由系统主动触发。

⚙️ Claude Code 路径

Claude Code 用状态切换——权限值划边界，计划文件传意图，不拆组件、不新增机制。

两种落地思路，解决的是同一个问题：让 Agent 在长任务里跑得远还不跑偏。
